# Verse8 gameserver 릴레이 지연 리포트 (실시간 멀티 렉)

> Verse8 팀 전달용. 게임: `hy.tae90/basic-git-repo` (V8 JETFALL, 실시간 호스트권위 멀티 슈터).
> 클라이언트 넷코드는 우리가 최적화 완료 — 남은 지배 원인은 **agent8 릴레이 왕복 지연**입니다.

## 한 줄 요약
실시간 멀티 매치의 렉 지배 원인은 게임 로직이 아니라 **릴레이 왕복 지연이 크고 편차가 극심**한 것입니다.
콜드스타트/스케일 정황이 보여, 인프라 설정(웜keep·리전)으로 개선 여지가 있는지 문의드립니다.

## 측정치 (자동화 2-클라이언트, 매치 진행 중, 백엔드 asia-northeast3)
- `remoteFunction('now')` 왕복(서버 실행 포함): **평균 400~640ms, 스파이크 1,830ms**
- 다른 세션 측정: **70~170ms** — 즉 **편차가 매우 큼**(안정적이지 않음)
- 스냅샷/입력/relay가 전부 remoteFunction 경로라 이 왕복이 곧 게임 지연 바닥값

## 왜 치명적인가 (구조)
호스트 권위 + 릴레이 구조에서 한 플레이어의 동작이 상대 화면에 반영되기까지:
`입력 → (릴레이 1홉) → 호스트 시뮬 → (릴레이 1홉) → 관측자` = **릴레이 2홉**.
왕복 400ms면 상대 동작이 ~0.4초 지연, 스파이크 시 순간 프리즈. 클라에서 예측(로컬)·보간(원격)으로
체감을 마스킹했지만 **바닥 지연 자체는 클라 코드로 못 줄입니다**.

## 문의/요청
1. asia-northeast3 게임서버 릴레이의 왕복 지연 목표치(p50/p95/p99)가 있나요? 위 수치는 정상 범위인가요?
2. **Cloud Run min-instances(웜keep)** 등으로 콜드스타트 스파이크(1.8s)를 줄일 수 있나요? scale-to-zero가 원인일까요?
3. `remoteFunction`(서버 isolated-vm 실행) vs 순수 `broadcastToRoom` 브로드캐스트의 지연 차이가 있나요?
   실시간 게임용으로 권장되는 **저지연 경로/옵션**이 있나요? (relayHot throttle 외)
4. 한국 외 플레이어 대비 **멀티리전/엣지 릴레이** 계획이 있나요?
5. 방/매치별로 지연 모니터링 지표를 개발자가 볼 방법이 있나요?

## 참고: 우리가 이미 한 클라이언트 최적화
- 전송 주기를 relayHot throttle(50ms)에 정렬(균일 20Hz, 드롭 0)
- 탄환 배치 전송(호출 캡 회피)
- 로컬 플레이어 예측(자기 캐릭터는 지연 무관 즉시 반응) + 원격 보간/반전 스냅
- 호스트 백그라운드 탭 프리즈 방지(Web Worker 시뮬 시계)
→ 이 위에서 남는 렉은 순수 릴레이 왕복 지연입니다.
